# 환경 관리

환경 페이지에서 시스템 관리자가 AICE 환경을 등록, 조회, 편집,
삭제하고, 고객-환경 매핑을 관리하며, 신뢰 레지스트리 키를
관리할 수 있습니다. 관리자 사이드바에서 **환경**을 클릭하여
열 수 있습니다.

이 페이지를 조회하려면 `aice-environments:read` 권한이 있는
시스템 관리자여야 합니다. 환경을 생성, 편집, 삭제하고 고객
매핑을 관리하려면 `aice-environments:write` 권한이 필요합니다.
신뢰 레지스트리 작업에는 `trust-registry:read`와
`trust-registry:write` 권한이 필요합니다.

![환경 페이지](../assets/admin-environments-table.png)

## 환경 테이블

테이블에 시스템의 모든 AICE 환경이 표시됩니다. 각 행에는 다음
정보가 표시됩니다:

- **AICE ID** — 환경의 고유 식별자.
- **이름** — 환경의 표시 이름, 선택적 설명이 아래에 표시됩니다.
- **상태** — 활성, 정지, 비활성 중 하나.
- **고객** — 이 환경에 연결된 고객 수. 숫자를 클릭하면 매핑을
    관리할 수 있습니다.
- **키** — 등록된 신뢰 레지스트리 키 수. 숫자를 클릭하면 키를
    관리할 수 있습니다.
- **작업** — 편집 및 삭제 버튼.

## 환경 생성

1. 오른쪽 상단의 **환경 생성** 버튼을 클릭합니다.
2. 필수 항목을 입력합니다:
    - **AICE ID** — 고유한 영숫자 식별자(하이픈 및 밑줄 허용).
    - **이름** — 환경의 표시 이름.
    - **설명** — 선택 사항.
    - **상태** — 초기 상태(기본값: 활성).
3. 선택적으로 **신뢰 레지스트리 키**를 체크하여 환경과 함께
    키를 등록할 수 있습니다:
    - **발급자** — 토큰 발급자 식별자.
    - **키 ID (kid)** — 키 식별자.
    - **공개 키 (JWK)** — JWK JSON을 붙여넣습니다. 서버가
        RFC 7638 SHA-256 JWK 지문(Thumbprint)을 다시 계산하여
        두 가지 형식으로 표시합니다(아래
        [JWK 지문 확인](#jwk-지문-확인) 참조).
    - **키 설명** — 선택 사항.
    - **만료 일시** — 선택 가능한 하드 만료 시각. 허용 형식과
        동작은 [키 만료 정책](#키-만료-정책-expires_at)
        섹션을 참조하세요.
4. 표시된 지문을 aice-web-next에 표시된 값과 별도 채널로
    비교한 뒤, **지문이 일치함을 확인했습니다** 체크박스를
    토글합니다.
5. **환경 생성**을 클릭하여 제출합니다. 지문 확인이 체크되기
    전까지는 버튼이 비활성화됩니다.

![환경 생성 대화상자](../assets/admin-environments-create-dialog.png)

## 환경 편집

1. 작업 열의 **편집** 버튼을 클릭합니다.
2. 이름, 설명 또는 상태를 필요에 따라 수정합니다.
3. **저장**을 클릭하여 변경 사항을 적용합니다.

## 환경 삭제

1. 작업 열의 **삭제** 버튼을 클릭합니다.
2. 모든 고객 매핑과 신뢰 레지스트리 키가 영구적으로
    삭제된다는 경고와 함께 확인 대화상자가 나타납니다.
3. **삭제**를 클릭하여 확인합니다.

![삭제 확인 대화상자](../assets/admin-environments-delete-dialog.png)

삭제하면 환경 레코드, 모든 관련 고객 매핑, 모든 신뢰
레지스트리 키가 제거됩니다. 이 작업은 되돌릴 수 없습니다.

## 고객 매핑 관리

**고객** 열의 숫자를 클릭하면 고객 매핑 패널이 열립니다.

### 고객 연결

1. **고객 연결** 버튼을 클릭합니다.
2. 드롭다운에서 고객을 선택합니다. 이미 이 환경에 연결된
    고객은 표시되지 않습니다.
3. **고객 연결**을 클릭하여 확인합니다.

### 고객 연결 해제

1. 고객 옆의 **삭제** 버튼을 클릭합니다.
2. 대화상자에서 제거를 확인합니다.

## 신뢰 레지스트리 키 관리

**키** 열의 숫자를 클릭하면 신뢰 레지스트리 패널이 열립니다.

### 키 등록

1. **키 등록** 버튼을 클릭합니다.
2. 필수 항목을 입력합니다:
    - **발급자** — 토큰 발급자 식별자.
    - **키 ID (kid)** — 키 식별자.
    - **공개 키 (JWK)** — JWK JSON을 붙여넣습니다. 서버가
        RFC 7638 SHA-256 JWK 지문(Thumbprint)을 다시 계산하여
        두 가지 형식으로 표시합니다(아래
        [JWK 지문 확인](#jwk-지문-확인) 참조).
    - **키 설명** — 선택 사항.
    - **만료 일시** — 선택 가능한 하드 만료 시각. 자세한
        내용은 아래 [키 만료 정책](#키-만료-정책-expires_at)
        섹션을 참조하세요.
3. 표시된 지문을 aice-web-next에 표시된 값과 별도 채널로
    비교한 뒤, **지문이 일치함을 확인했습니다** 체크박스를
    토글합니다.
4. **키 등록**을 클릭하여 제출합니다. 지문 확인이 체크되기
    전까지는 버튼이 비활성화됩니다.

### JWK 지문 확인

환경 생성 또는 키 등록 폼에 JWK가 붙여넣어지면, 서버가 공개
키의 RFC 7638 SHA-256 지문을 독립적으로 다시 계산하여 두 가지
형식으로 표시합니다:

- **base64url (표준)** — 패딩 없이 43자. 검증 시 비교해야 할
    값입니다.
- **hex (4바이트 블록)** — 64개의 16진수 문자를 8자씩 8개
    블록으로 나누고 `:`로 구분합니다. 시각적·구두 비교를 돕기
    위해 함께 제공됩니다.

두 형식 모두 같은 32바이트 SHA-256 다이제스트를 인코딩하며,
표현 방식만 다릅니다. 둘 다 잘리지 않은 전체 값으로 표시되며
각 형식에는 **복사** 버튼이 있습니다.

aice-web-next 시스템 관리자는 JWK 자체와는 별도의 신뢰할 수
있는 채널을 통해 지문을 전달합니다. 그래야 공격자가 붙여넣기
도중 공개 키를 바꿔치기 하더라도 비교 대상 값까지 통제할 수
없습니다. 확인 체크박스를 토글하기 전에 양쪽 값을 한 글자씩
대조하세요.

JWK가 잘못되었거나 지원되지 않는 `kty`를 사용하는 경우, 지문
대신 오류가 표시되고, 확인 체크박스는 나타나지 않으며, 제출
버튼은 비활성화 상태로 유지됩니다.

확인 후 JWK 텍스트 영역을 수정하면 확인 상태가 해제되고,
이전에 표시된 지문이 사라지며, 제출 버튼이 다시
비활성화됩니다. 운영자는 새 지문이 표시될 때까지 기다린 후
다시 확인해야 폼을 제출할 수 있습니다.

서버에서 계산된 지문은 `trust_registry.key_registered` 감사
세부 정보에도 함께 기록되므로, 등록된 값을 사후에 확인할 수
있습니다.

### 키 만료 정책 (expires_at)

신뢰 레지스트리 키마다 선택적인 `expires_at` 필드가 있으며 두
가지 정책 중 하나를 따릅니다:

- **소프트 만료(기본값).** 등록 시 필드를 비워두면 운영자가
    수동으로 비활성화 또는 제거하기 전까지 키가 계속 신뢰
    상태로 유지됩니다. 회전은 정책에 의해 진행되며
    플랫폼이 강제하지 않습니다.
- **하드 만료.** `2026-05-05T12:00:00Z` 또는
    `2026-05-05T21:00:00+09:00`처럼 타임존이 명시된 ISO 8601
    일시를 입력합니다. 해당 시각이 지난 후에는 `enabled`
    플래그와 무관하게 이 키로 서명된 모든 브리지 요청이
    거부됩니다. 거부 판단은 모든 검증마다 새로 평가되므로
    캐시에 의해 지연되지 않습니다.

키 등록 UI는 일반 텍스트 입력 필드입니다. 운영자는 `Z` 또는
`±HH:MM`처럼 타임존이 명시된 ISO 8601 일시를 직접 입력해야
하며, 타임존이 없는 값(`2026-05-05T12:00:00`)이나 날짜만 있는
값은 브라우저에서 한 번, 서버에서 한 번 더 검증 오류로
거부됩니다.

과거 시각도 의도적으로 허용됩니다. `expires_at`을 과거로
설정하면 별도의 비활성화 단계 없이 침해된 키를 즉시 "소거"
할 수 있습니다.

신뢰 레지스트리 키 표는 키별 만료 거리에 따라 표시 색상을
변경합니다:

- **만료 없음 (소프트)** — 필드가 비어 있는 경우.
- 특정 날짜 — 만료까지 30일 이상 남은 경우(중립 색상).
- **노란색** — 만료까지 30일 이내.
- **빨간색** — 만료까지 7일 이내.
- **회색 "만료됨"** — `expires_at`을 지난 경우. 키는 여전히
    표에 표시되지만 이 키로 서명된 브리지 호출은 거부됩니다.

### 키 활성화 또는 비활성화

작업 열에서 **비활성화** 또는 **활성화**를 클릭하여 키의
상태를 전환합니다. 비활성화된 키는 토큰 검증에 사용되지
않습니다.

### 키 제거

1. 작업 열의 **제거** 버튼을 클릭합니다.
2. 대화상자에서 제거를 확인합니다.

키 제거는 영구적이며 되돌릴 수 없습니다.

## 감사 추적

환경 관리 작업은 감사 로그에 기록됩니다:

- **environment.created** — 환경이 등록된 경우.
- **environment.updated** — 환경 세부 정보가 변경된 경우.
- **environment.deleted** — 환경이 삭제된 경우.
- **environment.customer_linked** — 고객이 연결된 경우.
- **environment.customer_unlinked** — 고객 연결이 해제된
    경우.
- **trust_registry.key_registered** — 키가 등록된 경우.
- **trust_registry.key_disabled** — 키가 활성화 또는
    비활성화된 경우.
- **trust_registry.key_removed** — 키가 제거된 경우.

[감사 로그](audit-logs.md) 페이지에서 이 항목들을 확인할 수
있습니다.

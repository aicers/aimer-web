# 스토리 리프 재분석 백필

고객의 [기본 분석 모델](default-model.md)을 변경하면 그 변경은 **향후**
분석에만 적용되며, 기존 스토리 분석은 모두 **이전** 모델에 그대로
남습니다. 새 기본 모델 보고서는 여전히 완전한 것으로 표시되지만(커버리지를
위해 이전 모델 리프로 폴백함), 집계 점수는 처음에는 적은 수의 새 모델
리프로만 계산되므로 충분한 리프가 다시 분석되기 전까지 점수가 **낮게
표시**될 수 있습니다. 보고서의 커버리지 표시기가 이 일시적인 격차를
보여줍니다.

**스토리 리프 재분석 백필**은 이 전환 구간을 단축합니다. 고객의 기존
**스토리** 분석을 새 기본 모델로 다시 큐에 넣습니다. 이는 의도적이고
비용이 제한되며 운영자가 실행하는 작업으로, 기본값을 변경한다고 해서
**자동으로** 시작되지 않습니다.

> 이 페이지는 **스토리 리프**만 다룹니다. **이벤트** 리프 재분석과
> **보고서** 새로고침은 이 작업과 함께 제공되는 별도의 순차 작업이며,
> 보고서 새로고침은 스토리 리프 백필이 모두 처리된 후에만 실행됩니다.

## 실행 권한

백필은 실행 진입점인 고객별 기본 모델 컨트롤과 동일한 권한을 사용합니다.

- **시스템 관리자** — 모든 고객(관리자 화면).
- **분석가** — 배정된 고객(고객 설정 화면).

매니저와 사용자는 실행할 수 없습니다.

## 위치

기본 모델 변경이 성공하면 모델 변경 섹션에 **기존 데이터 다시 분석**이
제공되며, 해당 고객의 재분석 페이지가 열립니다. 이 페이지에 백필 컨트롤이
있습니다.

- 분석가 화면: **고객 설정 → 기존 데이터 다시 분석**.
- 관리자 화면: **관리자 → 고객 → (고객 선택) → 다시 분석**.

백필은 항상 고객의 **현재 유효 기본 모델**을 대상으로 합니다. 범위는
지정하지만 임의의 대상 모델을 선택할 수는 없습니다.

## 범위 지정

큐에 넣는 작업량을 제한하는 두 가지 큐 측 컨트롤이 있습니다.

- **최근 일수** — 가장 최근 활동이 이 일수 이내인 스토리만 대상으로
  합니다. 기본값은 보수적인 **7일**입니다. **전체 기간**을 선택하면 시간
  제한이 사라집니다(기본값 아님).
- **실행당 상한**(선택) — 한 번의 실행이 큐에 넣는 리프 수의 상한입니다.
  상한으로 제외된 리프는 **상한으로 제외됨**으로 보고되며, 다시 실행하면
  중단된 지점부터 이어집니다.

이들은 **큐(enqueue)** 측을 제한합니다. 큐를 처리하는 워커는 이미 독립적으로
스로틀링됩니다(틱당 고정 배치 크기와 최대 세대). 따라서 큰 실행도 한 번에
모델로 보내지 않고 여러 틱에 걸쳐 분산됩니다 — 최초 틱 분석 백로그가
의존하는 것과 동일한 스로틀링입니다.

## 비용 미리보기와 확인

큐에 넣기 전에 페이지는 실행 **미리보기**를 표시합니다. 대상 모델과
범주별 개수입니다. **다시 분석 시작** 버튼이 활성화되려면 명시적 확인에
체크해야 하며, 확인 없이는 실행이 진행되지 않습니다. 미리보기는 개수와
범위만 보고하며 **금액은 표시하지 않습니다**.

미리보기와 실행 후 진행 상황은 범위를 명확한 범주로 나누어 아무것도 조용히
누락되지 않도록 합니다.

| 범주 | 의미 |
| --- | --- |
| **다시 분석할 리프** | 새로 추가할 리프와 새 모델로 다시 큐에 넣을 실패 리프의 합. |
| **이미 최신**(`coalesced`) | 이미 새 모델로 분석되었거나 처리 중인 리프 — 그대로 둠. |
| **소스 변경 대기 중**(`skipped_dirty`) | 스토리 소스가 변경되어 정상 새로고침을 기다리는 중. 이중 큐를 피하기 위해 백필이 워커에 맡김. |
| **소스 삭제됨**(`source_unavailable`) | 스토리 소스가 보관 처리되거나 삭제됨 — 다시 분석할 수 없으므로 실패할 작업을 만들지 않고 건너뜀. |
| **상한으로 제외됨**(`cap_excluded`) | 실행당 상한으로 제외된 범위 내 리프. |

## 멱등성

같은 범위로 백필을 다시 실행해도 안전하며 이중 큐가 발생하지 않습니다.

- 새 모델로 이미 **큐/처리 중/완료**된 리프는 **병합(coalesce)**되어 그대로
  유지됩니다 — 중복 작업도, 세대 증가도 없습니다.
- **실패한** 리프는 동일한 세대로 **다시 큐에 넣어** 워커가 재시도합니다.
- 실제로 **없는** 리프만 새로 추가됩니다.

이는 세대를 무조건 증가시키는 단일 항목 **강제 재생성** 경로와 의도적으로
다릅니다.

## 진행 상황

페이지는 현재 범위에 대한 **진행 상황**을 보여줍니다. 새 모델로 아직
**남아 있는(outstanding)** 리프 수와 완전히 다시 분석된(**완료된**) 리프
수입니다. 리프는 새 모델 분석이 **완료**되고 소스가 변경 중이 아닐 때에만
완료로 집계됩니다. 새 모델 분석이 없거나, 큐에 있거나, 처리 중이거나,
**실패한** 리프는 범위를 미완료 상태로 유지합니다. 소스가 삭제된 스토리는
제외됩니다 — 다시 분석할 수 없으므로 진행을 막지 않습니다.

이 진행 신호는 이후 **보고서 새로고침**의 게이트이기도 합니다. 범위의 스토리
리프가 모두 처리되기 전까지 보고서는 새 모델로 새로고침되지 않으므로, 아직
다시 분석되지 않은 오래된 집합을 다시 집계하는 일이 없습니다.

## 비용 영향

리프를 다시 분석하면 리프당 모델을 한 번 호출하므로, 실행 비용은 큐에 넣는
리프 수에 비례합니다. 실행 범위를 제한하세요.

- 기본 최근 기간으로 시작하고 필요할 때만 넓히세요.
- 매우 큰 고객은 실행당 상한을 사용하고 반복 실행하여 사이에 워커가 큐를
  처리하도록 하세요.
- 백필은 전환 구간을 **단축**할 뿐입니다. 완료 전에도 보고서는 폴백
  커버리지로 완전하게 유지되므로, 모든 기존 데이터를 한 번에 다시 분석할
  필요는 없습니다.

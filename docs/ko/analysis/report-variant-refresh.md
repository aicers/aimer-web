# 모델 변경 후 정기 보고서 새로고침

고객의 [기본 분석 모델](default-model.md)을 변경한 뒤 기반 리프를 다시
분석하면([스토리](reanalyze-backfill.md), [이벤트](event-leaf-reanalysis.md)),
리프는 새 모델로 이동합니다. 하지만 기존 정기 **보고서**는 재생성되기
전까지 여전히 **이전 리프 집합을 집계**합니다. 정기 보고서의 대표 점수와
우선순위 등급은 해당 보고서에 선택된 스토리·이벤트 리프로부터 계산되므로,
보고서를 **새로고침**하기 전까지는 재분석된 리프가 반영되지 않습니다.

**보고서 변형 새로고침**은 범위가 지정된 정기 보고서 변형을 새 모델로
재생성하여 방금 재분석된 리프를 다시 집계합니다. 재분석 페이지의 **세
번째** 패널로, 스토리·이벤트 재분석 패널 다음에 있습니다.

이는 의도적인 운영자 트리거 작업입니다. 보고서 재생성은 모델 호출 예산을
사용하므로 기본 모델을 변경한다고 해서 **자동으로 시작되지 않습니다**.

> **스크린샷 자리표시자.** 패널은 고객 데이터에서 가져온 실시간 보고서
> 변형 개수를 표시합니다. 대표 데이터가 있는 스택을 사용할 수 있을 때
> 실제 데이터 캡처가 추가됩니다.

## 리프 재분석 이후에 실행하세요

보고서는 스토리·이벤트 리프를 **모두** 집계하므로, 해당 리프가 재분석되기
전에 새로고침하면 **이전** 리프 집합을 다시 집계할 뿐 아무 진전도 없습니다.
따라서 새로고침은 각 보고서를 스토리·이벤트 리프 재분석 완료 여부에 대해
**자동으로 확인**합니다.

- 각 보고서 변형은 아래에서 선택하는 최근 윈도우가 아니라 **그 보고서의
  기간 윈도우**(실시간 최근 24시간, 일간 1일, 주간 7일, 월간 1개월)에 대해
  **두** 리프 드레인 신호로 확인됩니다. 오래된 주간·월간 보고서는 최근
  윈도우를 한참 벗어난 리프를 집계할 수 있으므로, 게이트는 보고서의 전체
  집계 윈도우를 확인합니다.
- 해당 윈도우에서 스토리 **또는** 이벤트 리프가 아직 재분석되지 않은
  보고서는 **게이트됨**으로 보고되며 새로고침되지 **않습니다**. 리프
  재분석이 끝난 뒤 새로고침을 다시 실행하세요.

게이트는 코드로 강제됩니다 — 아직 이전 모델 상태인 리프 위에서 보고서를
새로고침할 수 없습니다.

## 실행 권한

| 역할 | 범위 |
| --- | --- |
| **시스템 관리자** | 모든 고객(관리자 재분석 페이지에서). |
| **분석가** | 배정된 고객(고객 설정 → 재분석에서). |

매니저와 사용자는 새로고침을 시작할 수 없습니다.

## 실행 방법

1. 고객의 기본 분석 모델을 변경하고 스토리·이벤트 리프를 재분석하세요(처음
   두 패널). 처리가 끝나면 재분석 페이지로 돌아옵니다.
2. **정기 보고서 새로고침** 패널에 **범위 미리보기**가 표시됩니다.
3. *(선택)* 확인 전에 범위를 조정하세요.
   - **대상 언어** — 새로고침이 대상으로 하는 보고서 언어. 보고서 변형은
     언어별입니다. 기본값은 인터페이스 언어입니다.
   - **최근 윈도우(일)** — 버킷 날짜 기준으로 범위에 포함되는 보고서
     **버킷**. 기본값은 **7일**입니다. (위에서 설명한 보고서별 드레인 게이트
     윈도우와는 별개입니다.)
   - **회당 한도** — 이번 실행이 새로고침하는 보고서 변형 수의 상한(선택).
     비워 두면 한도가 없습니다. 한도를 초과한 변형은 **제한됨**으로
     보고됩니다.
   - **시간대** — 정기 보고서는 시간대로 키가 지정되므로 시간대 변경 후 이전
     시간대 버킷이 공존할 수 있습니다. **비워 두면** 윈도우 내 **모든**
     시간대 변형을 새로고침하며(기본값), IANA 시간대 하나(예: `Asia/Seoul`)를
     입력하면 해당 시간대 변형만 대상으로 합니다. 미리보기와 최근 실행 요약에
     실행이 대상으로 한 시간대가 표시됩니다.
   - **기간** — 특정 보고서 기간(실시간/일간/주간/월간)으로 제한. 기본적으로
     모든 기간이 범위에 포함됩니다.

   미리보기 개수는 입력한 범위에 맞춰 갱신됩니다.
4. 미리보기 개수를 검토한 뒤 **보고서 변형 N개 새로고침**을 선택하고
   **확인**하세요. 새로고침이 실행되고 분류된 결과 개수가 표시됩니다.

확인은 **필수**입니다 — 확인하기 전까지 아무것도 실행되지 않습니다.
미리보기는 **개수와 범위만** 표시하며 비용 금액은 절대 표시하지 않습니다.
**대상 모델**은 고객의 새 기본값으로 고정됩니다.

## 범위와 기본값

실행은 전체 기간을 조용히 새로고침할 수 없도록 제한됩니다.

- **고객** — 모델이 변경된 고객. 실행은 단일 고객 단위입니다.
- **대상 변형** — 선택한 언어에 대한 고객의 새 기본 `(model)`.
- **시간대** — 기본적으로 윈도우 내 모든 시간대 변형. 선택적으로 단일 IANA
  시간대로 제한합니다. 보고서 변형은 `(tz, lang, model_name, model)`로 키가
  지정되므로 이는 변형별 시간대 축입니다.
- **최근 윈도우** — 기본적으로 최근 **7일**의 보고서 **버킷**. 이는 *어떤
  버킷*이 큐에 들어가는지를 제한하며, 각 보고서의 기간에서 도출되어 7일보다
  훨씬 길 수 있는 보고서별 드레인 게이트 윈도우와는 구별됩니다.
- **기간** — 기본적으로 네 가지 모두. 필요에 따라 제한합니다.
- **회당 한도** *(선택)* — 한 번의 실행이 새로고침하는 변형 수의 상한.
  한도를 초과한 변형은 **제한됨**으로 보고됩니다.

### 새로고침 실행 방식

새로고침은 새로운 분석 워커가 **아닙니다**. 새로고침되는 각 보고서는 보고서
작업의 **세대 증가**이며, 기존 정기 보고서 워커가 이를 처리하여 재생성합니다.
그 워커는 이미 틱당 제한(`ANALYSIS_JOB_BATCH_SIZE`)되고 변형당
상한(`ANALYSIS_MAX_GENERATION`, 기본 50)이 적용됩니다. 최근 윈도우 범위와
선택적 회당 한도는 그 위에서 큐 투입 버스트를 추가로 제한하여, 보고서
재생성 비용 버스트를 리프 재분석 버스트와 분리하고 그 이후에 두도록 합니다.

## 개수의 의미

미리보기와 실행 보고서는 동일한 분류를 사용하므로 조용히 누락되는 것이
없습니다.

| 분류 | 의미 |
| --- | --- |
| **새로고침 예정** | 새 모델로 재생성될 보고서 변형. |
| **게이트됨** | 변형의 스토리 또는 이벤트 리프가 해당 기간 윈도우에서 아직 재분석되지 않음. 건너뜀 — 리프 재분석 후 다시 실행하세요. |
| **이미 대기열/처리 중** | 변형에 대한 재생성이 이미 진행 중이라 이중 증가를 막기 위해 건너뜀. |
| **재생성 한도 도달** | 변형이 `ANALYSIS_MAX_GENERATION`에 도달함. 한도를 넘겨 증가시키지 않고 보고만 함. |
| **원본 없음** | 보고서 원본 상태가 보관 처리(보존 정책으로 삭제)되어 재생성할 수 없음. |
| **제한됨** | 회당 한도로 제외된 새로고침 가능 변형. 계속하려면 다시 실행하세요. |

### 멱등성

다시 실행해도 안전합니다. 범위가 지정된 각 변형은 실행당 **최대 한 번**
새로고침되며, 이미 대기열에 있거나 처리 중인 변형은 건너뜁니다. 따라서 두
번째 실행이 보고서를 이중으로 증가시키거나 이미 재생성 중인 보고서에 예산을
다시 소비하지 않습니다. 재생성 한도에 도달한 변형은 한도를 넘기지 않고
한도 도달로 보고됩니다.

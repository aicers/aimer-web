# 모델 변경 후 이벤트 리프 다시 분석

고객의 [기본 분석 모델](default-model.md)을 변경하면 그 변경은 **향후**
분석에만 적용되며 기존 결과는 그대로 유지됩니다. 고객의 기존 데이터를 새
모델로 반영하려면 운영자가 **범위가 지정된 비용 제한 다시 분석**을 실행할 수
있습니다. 이 문서는 그중 **이벤트 리프** 부분을 다룹니다.

주기 리포트의 집계 점수와 우선순위 등급은 리포트에 선택된 **스토리** 리프와
**이벤트** 리프 **양쪽**에서 계산됩니다. 기본 모델 변경 후에는 이미 새
모델로 분석된 리프만 리포트 모델 점수에 반영됩니다. 이벤트 리프를 새 모델로
다시 분석하면 커버리지가 완전해져서, 리포트 자체를 새로고침하면 집계가 다시
전체 집합에 대해 계산됩니다.

이는 운영자가 의도적으로 실행하는 작업입니다. 다시 분석은 모델 호출 예산을
소모하므로 기본 모델을 변경한다고 해서 **자동으로 시작되지 않습니다.**

> **스크린샷 자리 표시자.** 이 패널은 고객 데이터에서 가져온 실시간 이벤트
> 개수를 표시합니다. 대표 데이터가 있는 스택을 사용할 수 있게 되면 실제
> 데이터 캡처를 추가합니다.

## 실행 권한

| 역할 | 범위 |
| --- | --- |
| **시스템 관리자** | 모든 고객(관리자 다시 분석 페이지에서). |
| **분석가** | 배정된 고객(고객 설정 → 다시 분석에서). |

매니저와 사용자는 다시 분석을 실행할 수 없습니다.

## 실행 방법

1. 고객의 기본 분석 모델을 변경합니다. 변경에 성공하면 해당 섹션에
   **기존 데이터 다시 분석** 제안이 표시됩니다.
2. 그 제안에서 다시 분석 페이지를 엽니다. **이벤트 리프 다시 분석** 패널이
   최근 기간에 대한 **비용 미리보기**를 표시합니다.
3. *(선택)* 확인하기 전에 범위를 조정합니다.
   - **대상 언어** — 실행이 대상으로 삼는 리포트 리프의 언어. 리포트 변형은
     언어별로 구분되므로 한 번의 실행은 한 언어만 다시 분석합니다. 기본값은
     현재 인터페이스 언어이며, 다른 언어의 리프를 백필하려면 전환하세요.
     미리보기, 확인 창, 시작된 실행 모두 정확한 대상 언어를 표시하므로 절대
     암묵적으로 결정되지 않습니다.
   - **최근 기간(일)** — 리포트의 이벤트 시간 기준으로 실행이 거슬러 올라가는
     범위. 기본값은 **7일**입니다. 기간을 줄이면 더 적은 이벤트를 다시
     분석합니다.
   - **실행당 상한** — 이 실행이 다시 분석하는 이벤트 수의 선택적 상한.
     상한 없이 실행하려면 비워 둡니다. 상한을 초과한 이벤트는 `cap_excluded`로
     보고됩니다.

   미리보기 개수는 입력한 범위에 맞게 갱신되므로, 확정하기 전에 효과를 확인할
   수 있습니다.
4. 미리보기 개수를 검토한 뒤 **이벤트 리프 N건 다시 분석**을 선택하고
   **확인**합니다. 실행이 정확히 그 범위로 백그라운드에서 시작되고, 페이지에
   실시간 진행 상황과 **취소** 컨트롤이 표시됩니다. 실행이 종료 상태(완료,
   취소, 실패)에 도달한 뒤에도 페이지는 마지막 실행의 고정된 범위와 분류된
   결과 개수를 계속 표시하므로, 상한이 적용되었거나 취소된 실행도 감사할 수
   있습니다.

확인은 **필수**입니다. 확인하기 전까지 아무것도 실행되지 않습니다.
미리보기는 **개수와 범위만** 표시하며 비용 금액은 표시하지 않습니다.
**대상 모델**은 고객의 새 기본값 — 방금 변경한 모델 — 으로 고정되므로, 실행은
항상 그 새 모델을 *향해* 다시 분석합니다.

## 범위와 기본값

실행은 전체 기록을 조용히 다시 분석하지 않도록 제한됩니다.

- **고객** — 모델이 변경된 고객.
- **대상 변형** — 선택한 언어에 대한 고객의 새 기본 `(모델)`. 실행이 다시
  분석하여 도달하려는 변형입니다.
- **최근 기간** — 기본값은 최근 **7일**이며, 주기 리포트가 이벤트를 선택할
  때 사용하는 동일한 이벤트 시간 기준으로 측정됩니다. 이 기간 내에서
  **이미 분석이 존재하는** 이벤트만 범위에 포함됩니다(이 백필은 분석된 적
  없는 이벤트가 아니라 기존 리프를 다시 분석합니다).
- **실행당 상한** *(선택)* — 단일 실행이 다시 분석하는 이벤트 수의 상한.
  상한을 초과한 이벤트는 `cap_excluded`로 보고됩니다.

### 자체 속도 조절 비용 제어

이벤트 다시 분석은 동기식입니다 — 이벤트당 모델 호출 한 번 — 따라서
백그라운드 실행은 큐 워커에 의존하지 않고 **자체적으로 호출 속도를
조절**합니다. 고정된 간격으로 틱마다 소규모 배치를 다시 분석하므로 모델 호출
버스트가 제한됩니다. 최근 기간 범위와 선택적 실행당 상한이 총비용을 추가로
제한합니다. 처리량을 조정해야 하는 운영자를 위해 배치 크기와 간격을 설정할 수
있습니다(`EVENT_BACKFILL_BATCH_SIZE`, `EVENT_BACKFILL_POLL_INTERVAL_MS`).

## 개수의 의미

미리보기와 실행 보고서는 동일한 분류를 사용하므로 아무것도 조용히 누락되지
않습니다.

| 분류 | 의미 |
| --- | --- |
| **다시 분석 대상** | 아직 새 모델로 분석되지 않은 기존 이벤트 리프 — 실행이 다시 분석할 이벤트. |
| **이미 새 모델 적용됨** | 이미 대상 변형으로 분석됨. 집계되지만 다시 실행하지 않음. |
| **원본 없음** | 보존 정책으로 편집된 원본 이벤트가 제거되어 여기에서 다시 분석할 수 없음. 건너뜀 — 실패가 아님. |
| **상한으로 제외됨** | 실행당 상한으로 제외된 다시 분석 가능 이벤트. |
| **실패** | 오류가 발생한 다시 분석(예: 모델 사용 불가). 별도로 보고됨. |

**원본 없음** 이벤트는 저장된 데이터로 다시 분석할 수 없으므로(새 원시
이벤트 재수집은 별도의 aice-web-next 작업) 완료를 막지 않습니다.

### 멱등성

다시 실행해도 안전합니다. 대상 변형의 현재 리프가 이미 있는 이벤트는
건너뛰므로, 두 번째 실행이 작업을 중복하지 않고 이미 완료된 이벤트에 예산을
다시 쓰지 않습니다.

## 전체 다시 분석과의 관계

이벤트 리프 다시 분석은 세 단계 시퀀스의 한 부분입니다.

1. **스토리 리프**를 새 모델로 다시 분석합니다.
2. **이벤트 리프**를 새 모델로 다시 분석합니다 *(이 문서)*.
3. 스토리 리프와 이벤트 리프 범위가 **양쪽 모두** 완전히 다시 분석(드레인)되면
   **리포트**를 새로고침합니다.

리포트 새로고침은 두 리프 다시 분석이 모두 드레인될 때까지 대기하므로,
리포트는 기반 리프가 새 모델로 완전해진 후에만 다시 계산됩니다. 원본 없음
이벤트는 드레인 검사에서 제외되므로, 보존 정책으로 제거된 이벤트가
새로고침을 무기한 막지 않습니다.

## 비용 영향

다시 분석되는 각 이벤트는 모델 호출을 한 번 수행합니다. 따라서 실행 비용은
미리보기의 **다시 분석 대상** 개수에 비례합니다 — 확인하기 전에 검토하세요.
비용을 제한하려면 더 짧은 기간이나 실행당 상한을 사용하고, 실행이 속도
조절되어 비용이 한 번에 발생하지 않고 시간에 걸쳐 분산된다는 점을 기억하세요.
